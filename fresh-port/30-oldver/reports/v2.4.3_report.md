## 1. 加解密 key / 算法对比(P1)— **两版完全相同**

### 1.1 整数 XOR(CryptUtils encryptInt:/decryptInt:)
**完全相同的指令序列**,key = `0x01011011`:

```
; 2.4.3 encryptInt: 0xf0f88   /   5.5.0 encryptInt: 0x124b28  (decryptInt: 紧随其后,逻辑一致)
movw r0, #0x1011
movt r0, #0x101        ; r0 = 0x01011011
eors r0, r2            ; r0 = 0x01011011 ^ input
bx   lr
```
encryptInt: 与 decryptInt: 是同一个 XOR(对合运算),两版二进制字节级一致。**确认与已知 5.5.0 `0x01011011` 一致,未变。**

### 1.2 AES key(GameData getEncrypKey)— **两版还原出同一把 32 字符 ASCII key**
`getEncrypKey` 用一连串 `movs rX,#imm` + `strb.w rX,[sp,#off]` 在栈上逐字节拼出 key 字符串(典型反 strings 手法)。用 `/tmp/recover_key.py` 跟踪寄存器立即数还原 sp[0..0x1f]:

- 2.4.3 `getEncrypKey` @ `0x4d448` →
- 5.5.0 `getEncrypKey` @ `0x6e3d8` →
- **两者都还原出 ASCII:`39653543fa0d66aa04c88bfe0d725d04`**(32 字符,本身像一个 MD5 hex)
- HEX: `3339363533353433666130643636616130346338386266653064373235643034`

### 1.3 AES 模式 / 参数(CryptUtils doCipher:key:context:options: + _CCCrypt)
反汇编 `decryptData:withKey:`(2.4.3 `0xf0d70` / 5.5.0 `0x1248f8`):两版都 `movs r1,#0x3` → **options = 3 = kCCOptionPKCS7Padding(1) | kCCOptionECBMode(2)**。

`decrypt:key:options:`(2.4.3 `0xf0dd4`):`movs r4,#0x1` = kCCDecrypt;`encrypt:` 用 `#0x0` = kCCEncrypt。

`doCipher` @2.4.3 `0xf0e04` 调 `_CCCrypt`(`0xf0eec`),参数区(`0xf0ecc`):
```
movs r2, #0x10 ; str r2,[sp]    -> keyLength = 0x10 = 16  => AES-128
movs r2, #0x0  ; str r2,[sp+4]  -> iv = NULL              => ECB
; r1=0 = kCCAlgorithmAES ; op 来自调用方(decrypt=1)
```

### 1.4 最终可复现的解密公式(touchHLE 必须照此实现)
```
AES-128-ECB, PKCS7 padding, IV = NULL
key (16 bytes) = ASCII "39653543fa0d66aa"   (即 32 字符 key 字符串的前 16 个 ASCII 字节)
```
实测:用 **前 16 字节** AES-128 解 `levelupHV.dat` → `bplist00`(成功);用全 32 字节 AES-256 → 乱码(失败)。**所以是 AES-128,只取 key 串前 16 ASCII。** 5.5.0 的 `decryptData:withKey:` options 同为 3,key 串相同 → **5.5.0 的 .dat 用这套同样能解**。

## 5. UserInfoData 等级处理差异 —— **5.5.0 把等级 XOR 混淆存储(2.4.3 不混淆)**

`UserInfoData` 与等级/经验相关方法对比:
| 方法 | 2.4.3 | 5.5.0 |
|------|------|------|
| `addXp:` | `0x93100` | `0xbb040` |
| `checkUpgrade` | `0x929b4` | `0xba754` |
| `getUpgradeReward:nextLevel:` | `0x9298c` | `0xba72c` |
| `curLevel`(getter) | `0x95584` **纯 ivar 读** | `0xbaffc` **读 ivar 后再 msgSend 变换**(非纯 getter) |
| `setCurLevel:` | `0x95594` | `0xbd550` |
| **`encryptCurLevel`** | **无** | **`0xbb030`(纯 ivar 读,返回原始混淆值)** |
| **`setNewCurLevel:`** | **无** | **`0xbc184`(纯 ivar 写)** |
| `xp`/`setXp:` | 有 | 有 |
| `upgradePercent`/`setUpgradePercent:` | 有 | 有 |

反汇编证据:
```
; 5.5.0 curLevel 0xbaffc —— 不是纯 getter:
ldr r2,[r0, <ivar>] ; 取存储的(混淆)level
... ; 载入 classref + selref
blx _objc_msgSend   ; 经一次方法调用变换后返回 => 解混淆(XOR decryptInt:)

; 5.5.0 encryptCurLevel 0xbb030 —— 纯 ivar getter,返回原始混淆值:
ldr r0,[r0, <ivar>]
bx  lr
```
**解读**:5.5.0 把 `curLevel` 以 XOR(`0x01011011`,见 §1.1)混淆后存 ivar,`encryptCurLevel` 取原始混淆值,`curLevel` getter 负责解混淆,`setNewCurLevel:` 写原始值。2.4.3 的 `curLevel` 是裸 ivar,没有这层。
> **对「等级不涨」的怀疑点**:若 5.5.0 在某处比较/写回时混用了「混淆值」与「明文值」,或解档时 `setNewCurLevel:`(写原始)与 getter 解混淆不配套,会导致等级显示/判定错乱。排查 5.5.0 时应同时盯 `curLevel`/`encryptCurLevel`/`setNewCurLevel:`/`checkUpgrade` 四者的取值口径是否一致(都走解混淆还是都走原始)。

## 6. 存档格式 UserInfoData initWithCoder:(P6)

`initWithCoder:`(2.4.3 `0x91c7c` / 5.5.0 `0xb99f4`)用到的解码/setter(用 `/tmp/resolve_sel2.py` 解析):

**2.4.3**:`decodeIntegerForKey:`, `decodeDoubleForKey:` + `setUserId:`/`setXp:`/`setNewVipGold:`/`setAvailableWorkers:`/`setIconIndex:`/`setNextStorySectionId:`/`setCurQuestResult:`,过程中调 `encryptInt:`(说明 **2.4.3 也用 XOR 混淆部分整型字段**,如 vipGold)。

**5.5.0**:`decodeIntegerForKey:`, **`decodeIntForKey:`(新增)**, `decodeDoubleForKey:` + `setUserId:`/`setNewVipGold:`/`setAvailableWorkers:`/`setIconIndex:`/`setNextStorySectionId:`/`setCurQuestResult:`/**`setGold:`(新增)**/**`setNewCurLevel:`(新增)**。

**差异小结(对解档崩溃/兼容性)**:
- 5.5.0 比 2.4.3 多解码 **`setNewCurLevel:`(等级,混淆存储)** 与 **`setGold:`**,且多用 `decodeIntForKey:`(除 `decodeIntegerForKey:` 外)。
- 5.5.0 的 `UserInfoData` 自身只用 Int/Integer/Double 三种解码;此前记录的「解档崩溃需补 Real/Bool/Date 臂」应来自**存档对象图里的其他类**(MapData/Quest/数组元素),而非 UserInfoData 本体——与本次观察一致。
- 字段 forKey: 的真实 key 字符串多为单字符/加密短名(`'^'`/`'_'` 等),非可读英文,touchHLE 侧按原样 byte 透传即可。

## 7. NetworkManager(原 P5)—— 真 socket 层,无干净离线总开关

2.4.3 `NetworkManager`(237 方法)是基于 `AsyncSocket` 的真实长连接(`onSocketDidDisconnect:`、`connectFirstInThisOpen`、`checkPacketDataSourceWithData:length:`、各种 `addXxxToServer`/`sendInfoToServer`)。**没有** `useLocalData`/`isOffline`/`mockResponse` 这类一键 mock 开关。

但存在可利用的离线相关方法:
- `getLocalUserAndMapInfo` —— 从本地取用户+地图信息(离线读档钩子)。
- `parseFriendInfoData:pos:len:islocal:` —— **带 `islocal:` 形参**,有「本地来源」解析分支。
- `parseVisitedStatisticsInfoForLocalUser:pos:len:header:`、`updateLocalAcceptedNotifyQuestData`/`updateLocalFinishedNotifyQuestData`/`updateLocalUnrewardNotifyQuestData`、`updateLocalDataDeviceId`、`isFileCanBeWriteInLocal`、`showMultiUserLoginWithLocalID`、`sendInfoToServerWithoutSaveToLocal`(反向:存本地不发服务器)。
- 连接状态:`isConnected`、`checkNetWorking`、`disconnect`、`reconnectUsingNewHD`、`checkTimeOut`。

> touchHLE 伪造服务器的现实路径:不是找一个 mock 开关,而是 **在 socket 层之上**直接喂数据——优先走 `getLocalUserAndMapInfo` + `loadFromLocal` 让游戏从本地档进入,并对必须的 server 包(登录 ACK、商店购买确认等)在 `connection:didReceiveData:`/`checkPacketDataSourceWithData:length:` 处注入伪造响应。命令分发在 `commandController`/`changeStateTo:withMessage:`。
