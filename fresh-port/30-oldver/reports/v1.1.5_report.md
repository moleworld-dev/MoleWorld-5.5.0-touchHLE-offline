# 摩尔庄园 1.1.5 (2012) 逆向报告 — 面向 5.5.0 离线移植

- 目标二进制: `30-oldver/v1.1.5/Payload/Mole's World.app/Mole's World`
  - Mach-O **universal: armv6 + armv7**,cryptid 0(已脱壳),老 ObjC runtime,Xcode 4.2 / iOS SDK 5.0,`CFBundleVersion = 1.1.5`,`com.taomee.MoleWorld`。
  - 全部用 `otool -arch armv7 -oV / -tvV` 导出(659 个 ObjC 类,13.6 万行类元数据,95 万行反汇编)。
- 对比基准: 5.5.0 `01-cracked/Payload/MoleWorld.app/MoleWorld`(thin armv7,符号已 strip,靠 `04-bridge/method_imp_map.tsv` 的地址定位)。

> **一句话结论**:1.1.5 是 5.5.0 的"精简同源前身",602 个核心 gameplay 类两版共有;**AES key 完全相同**;升级表/任务/商店/小游戏/存档在 1.1.5 里全是**纯本地 + 明文可解**,是 5.5.0 离线化的现成蓝图。最值钱的 5 件东西见文末。

## 1. 加解密 key / 算法 —— 已完全破解,且 1.1.5 == 5.5.0

### 1.1 AES key(★ 跨版本通用)
- 函数 `-[GameData getEncrypKey]`(1.1.5 imp `0x3c3d5`;5.5.0 imp `0x6e3d8`)。
- 用 `movs + strb.w` 把 **32 个 byte** 逐个写进栈缓冲区,再 `[NSData dataWithBytes:length:]` 返回。逐字节模拟还原:

  ```
  hex   : 33 39 36 35 33 35 34 33 66 61 30 64 36 36 61 61 30 34 63 38 38 62 66 65 30 64 37 32 35 64 30 34
  ascii : "39653543fa0d66aa04c88bfe0d725d04"   (32 字符的 hex 样式字符串)
  ```

- **1.1.5 与 5.5.0 的 `getEncrypKey` 字节完全一致**(两边都独立反汇编还原,byte-for-byte 相同)。

### 1.2 AES 算法细节(`CryptUtils`,实为 Jim Dovey 的 `NSData+CommonCrypto` 分类)
- 调用链:`+[CryptUtils decryptData:withKey:]` → `+[... decrypt:key:options:]` → `+[... doCipher:key:context:options:]` → `_CCCrypt`(CommonCrypto)。
- `encryptData/decryptData` 传 **`options = 3`** = `kCCOptionPKCS7Padding(0x1) | kCCOptionECBMode(0x2)`。
- `doCipher` 内:`movs r3,#0x10`(block size 16)、对 key 做 `subdataWithRange{0,16}`(**只取前 16 字节做 key**),报错串 `"Key is too small for the cipher size (%d < %d)"`(NSData+CommonCrypto 签名串)。
- **结论**:文件加密 = **AES-128-ECB + PKCS7**,**有效 key = `getEncrypKey` 返回的 32 字节里的前 16 字节** = ASCII `"39653543fa0d66aa"`。

### 1.3 实测验证(用前 16 字节解密,得 Apple bplist)
| 文件(1.1.5) | 解密结果 |
|---|---|
| `property.dat` | `bplist00` 明文 ✔ |
| `farmquest.dat` | `bplist00` 明文 ✔ |
| `timequest.dat` | `bplist00` 明文 ✔ |
| `113_0.dat`/`114_0.dat` | `bplist00` 明文 ✔ |
| `xiaotulv_userinfo.dat` | `bplist00`(NSKeyedArchiver)✔ |

**用同一 key 解 5.5.0 的 `114_0.dat`/`levelupHV.dat`/`113_0.dat` 也全部成功** → 加密体系跨所有版本不变。

### 1.4 XOR int 加密(`CryptUtils encryptInt:/decryptInt:`)—— 这是 5.5.0 新增,1.1.5 没有
- 5.5.0 `CryptUtils` 有 `encryptInt:`(`0x124b28`)、`decryptInt:`(`0x124b34`),即备忘里的 XOR `0x01011011`。
- **1.1.5 的 `CryptUtils` 没有这两个方法**(只有 md5/sha1/xor::/encryptData/decryptData/encrypt/decrypt/doCipher 共 12 个)。所以"对整型做 XOR 混淆"是 5.5.0 才加的反作弊层。

## 2. 升级表 / 等级不涨 bug —— 锁定:**不是解密失败,是逻辑**

### 2.1 1.1.5 升级表 = `114_0.dat`(不是 `levelupHV.dat`)
- `-[GameData loadUpgradeXP]`(1.1.5 `0x3d741`)流程(已逐指令解析):
  `mainBundle pathForResource:@"114_0.dat"` → `dataWithContentsOfFile:` → `getEncrypKey` → `decryptData:withKey:` → `dictionaryWithData:`(解析 bplist)→ `setDictionary:` 存进 `upgradeXPs`(`NSMutableDictionary`,属性 `upgradeXPs_`)。
- 解出的 `114_0.dat` 是 **`{"levelN": xp阈值}`** 字典:
  ```
  level1=50, level2=150, level3=350, level4=950, level5=1750, level6=2950,
  level7=4750, level8=7450, level9=11350, level10=17350, ... level36=7354250, level37=8354250
  ```

### 2.2 升级判定逻辑(1.1.5,可工作的参考实现)
- 链:`-[UserInfoData addXp:]`(`0x76d20`)→ `-[UserInfoData checkUpgrade]`(`0x766f2`)。
- `checkUpgrade` 关键:`[GameData sharedInstance].upgradeXPs` → `count` → 循环里 `[NSString stringWithFormat:@"level%d", lvl]` → `[upgradeXPs objectForKey:@"levelN"]` → 比较当前累计 xp。**键格式硬编码 `@"level%d"`**。

### 2.3 5.5.0 同链对比(地址来自 method_imp_map.tsv,已反汇编确认)
- `GameData loadUpgradeXP`(`0x6fb98`):**也读 `114_0.dat`**,同样 `getEncrypKey`+`decryptData:withKey:`+`dictionaryWithData:`+`setDictionary:` → `upgradeXPs`。
- `UserInfoData checkUpgrade`(`0xba754`):同样 `upgradeXPs.count` → `stringWithFormat:` → `objectForKey:` → `unsignedIntegerValue` 比较;**但额外调用了 `[CryptUtils decryptInt:]`**(对某个 check/阈值字段做 XOR)。
- `UserInfoData addXp:`(`0xbb040`):入口先 `CFAbsoluteTimeGetCurrent`,调用 `decryptInt:`,再 `upgradeXPs.count`、`checkUpgrade`、`calculatePercent`、`WrapperManager updateUserInfoView:`/`accumulateXP`。
- `WrapperManager addXp:`(`0x261970`)是更上层入口(经验链 `WrapperManager addXp: → UserInfoData addXp: → checkUpgrade`)。
- `xp`/`setXp:` getter/setter 两版都是**裸 ivar 访问(无 XOR)**:1.1.5 `0x78bb4`/`0x78bc4`;5.5.0 `0xbd5e0`/`0xbd5f0`。

### 2.4 对 5.5.0 "等级不涨"的判断(★ 纠正旧假设)
- **旧假设(levelupHV.dat 解密失败)是错的**:`levelupHV.dat` 用通用 key 能正常解密,而且它根本**不是等级 XP 表**——它是**建筑/餐厅升级表**(按 `ID` 如 30002/30101 索引,字段 `xp_reward / gold_reward / required_build_value / mole_upper_limit_count / upgrade_gold_cost / time_cost`)。
- 真正的等级 XP 表是 `114_0.dat`(`level%d`→阈值),5.5.0 bundle 里**存在且能解密**。
- 既然表能解、xp 是裸存,**5.5.0 等级不涨的根因落在 `addXp:`/`checkUpgrade` 的算术或反作弊短路**(`decryptInt:` 那条新增分支,或 `isHackData`/`checkUserinfoMd5:` 把升级判定吞掉)。下一步应在 touchHLE 里 hook `UserInfoData checkUpgrade`(`0xba754`)打印 `xp` 与 `objectForKey:@"levelN"` 的返回值,对比 1.1.5 的干净逻辑即可定位。

## 4. 存档格式(★ 比 5.5.0 简单,且有现成模板存档)

### 4.1 运行时存档(写入 Documents)
- 用户数据:`-[GameData saveUserInfoData]/loadUserInfoData` → 文件名 **`userinfo.dat`**。
- 地图数据:`-[GameData saveMapData]/loadMapData/saveDefaultMapData` → 文件名 **`map.dat`**。
- 编码:`-[GameData encodeLocalMapData]` = `NSKeyedArchiver archivedDataWithRootObject:` → `encryptData:withKey:`(AES-128-ECB,同一 key)。解档逆向即可。
- 默认地图模板:`-[GameData loadMapdataFromResource:]` 读 bundle 内 **`1.dat`**(本 bundle 未附带,可能随包变体)。

### 4.2 模板/默认存档(bundle 内,可直接用作离线初始档)★
- **`xiaotulv_userinfo.dat`**(AES 加密)解出一份完整 `UserInfoData` NSKeyedArchiver,实测是一个 **37 级账号**:
  - 字段:`userId(8044361)`, `username("Saturday")`, `gold(252548)`, `vipGold(1070)`, `xp(7425942)`, `curLevel(37)`, `upgradePercent`, `iconIndex(19)`, `curQuestId(15)`, `nextQuestId(16)`, `curQuestResult`, `nextStorySectionId`, `availableWorkers(7)`, `totalWorkers(73)`, `totalRooms(31)`, `mapExtend(287)`, `achieveUnlock`, `lastCoolDownTime`, `npcId`, `npcs`(`NpcData` 数组)。
  - 注:`xp=7425942` 落在 L36(7354250)与 L37(8354250)之间,与 `curLevel=37` 自洽;**xp/curLevel 在档里是明文整型,无 XOR**。
- **`xiaotulv_map.dat`**(★ **未加密**,裸 `bplist00` NSKeyedArchiver,3444 个对象):
  - 类:`TMMapDataBase` / `TMMapDataBuilding` / `TMMapDataFarm` / `TMMapDataSpacials` + `NSMutableDictionary`/`NSMutableArray`。
  - 字段:`baseTile, objectId, objectState, buildingState, farmState, cropId, coolingTime, beginTime, gameCoolTime, property, isFlip`。
  - 这 4 个 `TMMapData*` 类**两版共有**,所以这份地图存档结构对 5.5.0 基本通用。

### 4.3 与 5.5.0 对比
- 核心 archive 类(`UserInfoData`/`NpcData`/`TMMapData*`)两版同名共有 → 存档结构同源、字段更少。
- 5.5.0 在外面多包了一层反作弊(见 §5),1.1.5 没有。

## 5. 反作弊 —— 1.1.5 基本"裸奔",5.5.0 才加的

| 项 | 1.1.5 | 5.5.0 |
|---|---|---|
| `GameData md5Check:md5:` | **空 stub**(`bx lr` 直接返回,imp `0x3c4a4`)| 真实实现 `0x6e4c4` |
| `GameData isHackData` | **不存在** | 存在 `0x8b5e8` |
| `GameData checkUserinfoMd5:` / `CheckUserInfoData:` | **不存在** | `0x753b8` / `0x754c0` |
| `CryptUtils encryptInt:/decryptInt:`(XOR 混淆)| **不存在** | `0x124b28` / `0x124b34` |
| `NewSceneData`/`NewSceneUserInfoData` 的 `isHackData`/`checkUserinfoMd5:` | **不存在(无 NewScene)** | 存在 |

> 启示:1.1.5 证明这游戏**不带反作弊也能正常跑存档**。在 touchHLE 离线版里,把 5.5.0 的 `isHackData`/`checkUserinfoMd5:`/`CheckUserInfoData:` stub 成"永远合法",以及确认 `decryptInt:` 不破坏升级判定,是安全的(和 1.1.5 行为一致)。

## 6. ObjC 类总览与对比

- 1.1.5:**659 类**;5.5.0:**1968 类**;**共有 602 类**(几乎所有 1.1.5 gameplay 类都在 5.5.0 里)。
- **1.1.5 有、5.5.0 没有的 gameplay 类(共 57,挑重点)**:
  - SQLite 直连:`DBConnection`、`Statement`、`Status`(1.1.5 自带轻量 SQLite 层;5.5.0 改用别的持久化)。
  - 旧商店:`ShopItems`、`ShopLayer`、`StoreItems`、`StoreViewLayer`、`StoreCell`(5.5.0 重做了商城类名)。
  - 旧登录/账号:`Login`...、`LoginHTTPConnectManager`、`OAuthController`、`OAuthEngine`、`ApplyIDDialog`、`SetPasswordDialog`、`PasswordModifyDialog`、`User`、`UserIDListView`、`Stopwatch`、`DummyGapStatus`。
  - 其余多为社区/反馈/客服(`Feedback*`/`Fetch*`/`QAThread`/`Comment`/`Draft`/`DirectMessage`)——离线无关。
- 方向上:5.5.0 多出的 1300+ 类是新增内容(新场景 `NewScene*`、黄金岛/加勒比、更多活动/小游戏、反作弊)。**离线移植的"减法基线"应以 1.1.5 的 602 共有类为准**。

## 8. 工作脚本(复现用,均在 /tmp)
- `/tmp/parse_objc.py`:解析 `otool -oV` 导出,按类列方法。
- `/tmp/resolver.py`(1.1.5)/`/tmp/resolver550.py`(5.5.0):反汇编里 `movw/movt + add pc + ldr` 字面量(selector / NSString / classref)解析器。
- `/tmp/keysim3.py`:模拟 `getEncrypKey` 的 `strb` 序列还原 32 字节 key。
- 解密一行流:`AES.new(b"39653543fa0d66aa", AES.MODE_ECB).decrypt(data)` 去 PKCS7,即得 bplist。
